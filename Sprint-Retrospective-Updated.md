# EC360 Project Sprint Retrospective\n\n**Team:** RevMatch Paper Review System\n**Sprint:** Core User Stories (Registration, Dashboards, Paper Management)\n\n## What Went Well ✅\n\n- Universal Registration/Login for all roles (reviewer/author/editor)\n  - Password hashing (bcrypt), JWT auth, role-based redirection ✓\n- Reviewer Dashboard: View assigned papers w/ PDF links & expertise management ✓\n- Author Dashboard: Track paper status (color-coded) ✓\n- Auto reviewer assignment by expertise match ✓\n- Clean React UI w/ loading states & error handling\n\n## What Went Wrong ❌\n\nFile casing issue on Windows — User.js was initially saved as user.js which caused a compilation error. This happened because Windows is case-insensitive but Git and Node.js are not. It caused a delay of half a day before being resolved using git mv.\nMulter middleware misconfiguration — During EP-3, the file upload field name in the frontend did not match the backend upload.single() parameter, causing a MulterError: Unexpected field error. This was fixed by aligning the field name to file on both ends.\nMongoDB duplicate key error — While testing EP-4, submitting the same reviewer email twice caused an unhandled crash instead of a clean error message. A duplicate email check was added to the route to handle this gracefully.\nCORS error on first API call — When the React frontend first tried to call the Express backend, a CORS policy error appeared in the browser. This was resolved by adding the cors middleware in index.js.\n\n## Improvements (Priorities) 🚀\n\n1. **Review Submission Feature**\n   - New ReviewPaper.jsx + backend route\n   - Add comments/decision to Assignment model\n   - Update paper statuses automatically\n\n2. **Performance Monitoring**\n   - API timing middleware\n   - PDF optimization, caching\n   - Verify all 2s NFRs\n\n**Score: 7/10** - Solid foundation despite early bugs, missing review core
